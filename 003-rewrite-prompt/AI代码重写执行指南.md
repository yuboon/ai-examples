## 阶段一：代码分析与需求提取

### 步骤1.1：快速项目扫描

```bash
# 获取项目基本信息
find . -name "*.json" -o -name "*.yaml" | head -20
find . -name "*.js" -o -name "*.ts" -o -name "*.jsx" -o -name "*.tsx" | wc -l
find . -type f -name "package.json" -o -name "pom.xml" -o -name "requirements.txt"
```

**提供给AI的信息：**
```
项目路径: /path/to/old-project
项目类型: [观察后填写]
大概文件数: [统计后填写]
主要技术栈: [从配置文件识别]
```

### 步骤1.2：执行代码分析

**使用提示词版本1（完整分析）：**

1. 打开 `代码分析提示词.md`
2. 复制"提示词版本1：完整版"
3. 替换 `[你的项目路径]` 为实际路径
4. 发送给AI
5. 等待分析完成

**预期输出：**
- 完整的需求文档Markdown
- 包含所有功能规格
- 数据模型定义
- API规格（如适用）

### 步骤1.3：保存需求文档

```bash
# 创建文档目录
mkdir -p docs
cd docs

# 保存AI输出的需求文档
# 文件名: requirements.md
```

### 步骤1.4：补充缺失信息

如果AI分析结果不完整，针对性补充：

```markdown
## 补充分析请求

请进一步分析以下内容：

1. [缺失的功能模块]
2. [未识别的数据模型]
3. [需要澄清的业务逻辑]

请详细说明：
- 相关代码位置
- 具体实现逻辑
- 与其他模块的关系
```

## 阶段三：AI代码生成

### 步骤3.1：项目初始化

```bash
# 创建新项目目录
mkdir new-project
cd new-project

# 初始化基础项目
# 根据技术栈选择命令

# React/Vite
npm create vite@latest . -- --template react-ts

# Next.js
npx create-next-app@latest . --typescript --tailwind

# Vue
npm create vue@latest . --typescript

# Node.js
npm init -y
npm install -D typescript @types/node
```

### 步骤3.2：分阶段生成代码

**阶段1：基础设施**

```markdown
## 代码生成请求 - 阶段1

基于需求文档 `requirements.md`，请生成项目基础设施：

1. 目录结构（按文档第2.2节）
2. TypeScript配置
3. ESLint/Prettier配置
4. 环境变量定义
5. 基础类型定义（第3节数据模型）

请逐个生成文件，确保每个文件包含完整的代码。
```

**阶段2：数据层**

```markdown
## 代码生成请求 - 阶段2

请生成数据层代码：

1. 数据模型文件（types/models.ts）
2. API服务封装（services/api.ts）
3. 状态管理配置（stores/）
4. 工具函数（utils/）

参考需求文档第3、5、6节。
```

**阶段3：基础组件**

```markdown
## 代码生成请求 - 阶段3

请生成基础UI组件：

1. 按钮组件（components/Button.tsx）
2. 输入组件（components/Input.tsx）
3. 模态框组件（components/Modal.tsx）
4. 加载组件（components/Loading.tsx）

参考需求文档第6、7节中的组件规范。
```

**阶段4：功能实现（逐个功能）**

```markdown
## 代码生成请求 - 功能F001

请实现功能 F001: [功能名称]

参考需求文档第4.1节中的功能规格。

生成内容：
1. 页面组件（pages/[功能页面].tsx）
2. 相关服务函数（services/[功能].ts）
3. 相关类型定义（types/[功能].ts）
4. 单元测试（tests/[功能].test.ts）

确保：
- 遵循命名规范
- 包含错误处理
- 添加必要注释
- 实现所有业务规则
```

### 步骤3.3：增量生成策略

```markdown
# 增量生成最佳实践

1. **按依赖顺序生成**
   基础设施 → 数据层 → 基础组件 → 页面组件 → 业务逻辑

2. **每次生成一个完整单元**
   - 一个功能模块
   - 一个页面及其依赖
   - 一个组件及其测试

3. **生成后立即验证**
   - 代码能否编译
   - 类型是否正确
   - 导入是否有效

4. **保持上下文连续**
   - 在同一对话中持续生成
   - 定期提供项目当前状态
   - 明确已生成的内容
```

## 阶段四：代码验证与测试

# 目标：每模块 > 80%

## 常见问题与解决方案

### 问题1：AI分析不完整

**现象：** 需求文档缺少部分功能或规格

**解决方案：**
```markdown
## 补充分析请求

需求文档中缺少以下内容：
1. [具体缺失项]
2. [具体缺失项]

请分析以下代码文件并补充：
- 文件路径1
- 文件路径2

提供详细的规格说明。
```

### 问题2：生成的代码无法编译

**现象：** TypeScript错误、导入错误

**解决方案：**
```markdown
## 代码修复请求

生成的代码存在以下问题：

[粘贴错误信息]

请修复这些问题，确保：
1. 类型定义正确
2. 导入路径有效
3. 依赖关系正确
4. 代码符合规范
```

### 问题3：业务逻辑理解偏差

**现象：** 实现的代码与原始逻辑不符

**解决方案：**
```markdown
## 逻辑修正请求

当前实现与原始逻辑存在差异：

原始逻辑描述：
[描述正确的业务逻辑]

当前实现：
[粘贴当前代码]

请修正实现，确保：
1. 业务规则一致
2. 边界条件处理正确
3. 异常处理完整
```

### 问题4：依赖关系混乱

**现象：** 模块间循环依赖、导入错误

**解决方案：**
```markdown
## 依赖关系重构

当前项目存在依赖问题：

[描述问题]

请重新规划模块依赖关系：
1. 绘制依赖图
2. 识别循环依赖
3. 提出重构方案
4. 执行重构
```

### 问题5：技术栈迁移问题

**现象：** 旧技术栈特性在新技术栈中无法实现

**解决方案：**
```markdown
## 技术栈迁移咨询

我需要将以下功能从 [旧技术] 迁移到 [新技术]：

功能描述：[描述]
旧实现：[粘贴旧代码]

请提供：
1. 新技术栈中的等效实现
2. 必要的调整和妥协
3. 最佳实践建议
```

## 最佳实践

### 1. 文档管理

```bash
# 推荐的目录结构
new-project/
├── docs/
│   ├── requirements.md           # 需求文档
│   ├── api-spec.md               # API规格（如果复杂）
│   ├── data-models.md            # 数据模型（如果复杂）
│   └── changelog.md              # 变更记录
├── src/                          # 源代码
├── tests/                        # 测试
└── .ai-prompts/                  # AI提示词存档
    ├── analysis-phase1.md
    ├── analysis-phase2.md
    └── generation-F001.md
```

### 2. 版本控制

```bash
# 关键节点打标签
git tag -a v1.0.0-analysis -m "需求分析完成"
git tag -a v1.0.1-infra -m "基础设施生成完成"
git tag -a v1.0.2-data-layer -m "数据层生成完成"

# 功能分支
git checkout -b feature/F001
# 生成功能F001代码
git checkout main
git merge feature/F001
```

### 3. AI对话管理

```markdown
# 每个阶段使用新的对话上下文

## 对话1：代码分析
- 目标：提取完整需求文档
- 输入：代码路径 + 分析提示词
- 输出：requirements.md

## 对话2：基础设施生成
- 目标：生成项目基础结构
- 输入：requirements.md + 基础设施请求
- 输出：项目骨架代码

## 对话3-N：功能生成
- 目标：逐个实现功能
- 输入：requirements.md + 功能规格
- 输出：功能代码
```

### 4. 质量保证

```markdown
## AI生成代码的质量检查点

1. **生成时检查**
   - 类型安全（TypeScript编译通过）
   - 语法正确（ESLint无错误）
   - 代码规范（Prettier格式化）

2. **功能检查**
   - 对照需求文档逐项验证
   - 业务规则实现正确
   - 异常处理完整

3. **非功能检查**
   - 性能考虑
   - 安全考虑
   - 可维护性
   - 可测试性

4. **测试覆盖**
   - 单元测试覆盖率 > 80%
   - 关键路径有集成测试
   - 边界条件有测试
```

### 5. 迭代优化

```markdown
## 持续改进循环

1. **生成**
   使用AI生成代码

2. **验证**
   运行测试和检查

3. **修正**
   向AI反馈问题并修正

4. **优化**
   识别改进点并优化

5. **文档化**
   更新需求和文档

循环执行，直到满足质量标准。
```

## 总结

使用AI进行项目重写的成功关键：

1. **完整的需求分析** - 花时间做好代码分析
2. **结构化的文档** - 使用AI友好的格式
3. **分阶段的生成** - 不要一次性生成所有代码
4. **持续的质量检查** - 每个阶段都要验证
5. **迭代式优化** - 根据反馈不断改进

遵循本指南，可以高效地将旧项目迁移到新技术栈，同时保证代码质量和可维护性。
